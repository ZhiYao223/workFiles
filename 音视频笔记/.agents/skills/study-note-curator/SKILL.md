# 学习笔记整理

## 概述

使用这个 skill 维护 `学习清单.md`，把用户在聊天中学习到的问答内容整理成简洁、正确、适合入门者复习的笔记，并放到学习清单中对应的章节或问题下面。

## 核心流程

1. 定位项目笔记文件，优先使用当前工作区根目录下的 `学习清单.md`。
2. 根据用户消息判断当前学习目标，例如 `二、视频编解码基础，重点` 或 `1. 视频为什么需要压缩`。
3. 编辑前先阅读 `学习清单.md` 中相关章节。
4. 从当前对话中提炼答案，优先保证：
   - 对入门软件工程师来说是正确的；
   - 表达清楚，不追求过深；
   - 贴近多媒体芯片 SDK、验证、调试工作中会用到的内容；
   - 有能帮助用户在工作中识别概念的具体例子。
5. 把整理后的笔记写入 `学习清单.md` 中匹配的位置。
6. 保留原始学习清单条目，在条目下面补充笔记，不要直接替换原条目。
7. 保持 Markdown 标题层级和编号格式。
8. 写入前先压缩内容：把聊天中的解释改写成复习笔记，不要把聊天回答原样搬进去。

## 写入规则

当整理某个具体编号问题时，优先在章节的 `### 学习笔记` 下用 `#### 问题标题` 记录，不要在原始清单条目中插入大量内容：

```markdown
#### 1. 视频为什么需要压缩

视频必须压缩，因为原始视频数据量极大，直接存储、传输、处理都会占用大量带宽、存储和算力。编码器的作用就是利用画面内和帧与帧之间的冗余，把 YUV 图像帧压成 `.h264`、`.h265` 等码流；解码器再把码流还原成可显示或继续处理的图像帧。工作中常见的码率、画质、延迟、解码失败等问题，本质上都和这条压缩/解压链路有关。
```

如果同一个问题已经有笔记，把新内容合并进原段落或原有少量分点，不要重复创建多个小标题。

如果用户学习的是整个章节，则更新该章节的 `### 学习笔记` 部分，并按章节里的编号问题组织内容。

如果写入位置不完全明确，优先根据标题和问题文本做合理匹配。只有当两个或多个位置同样可能时，才简短询问用户确认。

## 笔记风格与篇幅控制

默认使用中文写笔记，除非用户明确要求其他语言。

默认把笔记写成“紧凑复习版”，不要写成聊天回答或教程讲义。优先使用 1 到 2 个自然段；只有当概念层级较多、确实需要拆分时，才使用少量分点。

篇幅参考：

1. 简单概念：80-150 字，1 段即可。
2. 中等概念：150-250 字，1-2 段，必要时最多 3 个分点。
3. 难概念或易混概念：250-400 字，可以使用少量分点或一个短代码块/流程图。
4. 单个小问题一般不要超过 400 字；只有用户明确要求详细整理时才放宽。

排版要求：

1. 避免固定套用 `一句话理解`、`关键点`、`工作中怎么用`、`容易混淆` 这类标签。
2. 避免每句话都单独换行；相近意思合并成一个自然段。
3. 避免大规模编号列表。使用分点时，每个问题通常控制在 2-4 点。
4. 避免空行过多。标题后直接写正文，段落之间保留一个空行即可。
5. 代码块、流程图只在能明显降低理解成本时使用；不要为了形式完整而加入。
6. 对同一章节内相近问题，可以合并表达，减少重复。

优先采用这种紧凑格式：

```markdown
#### 1. 视频为什么需要压缩

视频必须压缩，因为原始视频数据量极大，直接存储、传输、处理都会占用大量带宽、存储和算力。编码器会利用画面内和帧间的冗余，把 YUV 图像帧压成 `.h264`、`.h265` 等码流；解码器再把码流还原成图像帧。对岗位来说，理解压缩是为了后面看懂码率、画质、延迟、硬编硬解和 SDK 参数。
```

对于确实复杂的问题，可以使用这种有限分点格式：

```markdown
#### 2. 原始视频数据基础

原始视频通常可以理解为还没经过 H.264/H.265 压缩的一帧帧图像数据，岗位里最常见的是 YUV，而不是 RGB。需要重点记住几个参数：

1. 分辨率决定一帧有多少像素，例如 1920x1080。
2. 帧率决定每秒多少帧，例如 30fps。
3. 像素格式决定数据怎么排布，例如 YUV420、NV12、NV21。

这些参数如果和 SDK 或硬件要求不一致，常见结果就是花屏、绿屏、颜色异常或 buffer 大小计算错误。
```

无论长短，笔记都要保证用户之后在 Typora 里单独阅读时能看懂。

## 质量要求

应该做到：

1. 从入门者能理解的角度解释概念。
2. 把概念和用户目标岗位联系起来：H.264/H.265/JPEG、FFmpeg、SDK、FPGA 验证、回片验证、客户支持。
3. 笔记要实用，方便之后复习。
4. 相关时使用这些例子：原始 YUV、`.h264`、`.h265`、`.mp4`、码率、帧率、SPS/PPS、IDR、NV12、stride、解码器输入输出、SDK 参数。
5. 如果对话中的信息不足，要说明不确定性，不要硬编。

避免：

1. 堆砌标准文档级别的细节。
2. 写成过于学术的说明。
3. 添加没有依据的内容，除非是基础且稳定的背景知识。
4. 改写无关章节。
5. 删除用户已有笔记。
6. 把聊天中的完整回答直接搬进 Markdown。
7. 为每个小问题都套用同一套固定标签，造成视觉噪声。
8. 使用过多空行、过多列表、过多小标题，让笔记像 GPT 回答而不是复习资料。

## 示例流程

用户：`我要学习“二、视频编解码基础，重点”，先问：视频为什么需要压缩？`

先在聊天中给出入门解释。除非用户要求记录，否则暂时不编辑文件。

用户：`这一小节学完了，帮我整理进学习清单。`

从对话中提炼最终答案，并更新 `学习清单.md` 中 `## 二、视频编解码基础，重点` -> `1. 视频为什么需要压缩` 的位置。

用户：`把刚才关于 YUV/NV12 的内容也写进去。`

在同一章节中找到匹配条目，通常是 `2. 原始视频数据基础`，并把笔记合并进去。
