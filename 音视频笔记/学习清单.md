### 学习笔记

#### 1. 芯片多媒体系统软件岗位是做什么的

这个岗位的核心不是单纯写 App，也不是单纯做视频算法，而是让芯片里的视频/图像编解码硬件模块能被软件正确使用起来。芯片中会有视频编码器、视频解码器、JPEG 编解码器等硬件模块，软件需要配置输入输出地址、分辨率、编码格式、码率、启动停止等参数，再通过 SDK 封装成客户能调用的接口。

实际工作会围绕 FPGA 验证、回片验证、SDK 开发维护、Demo/Sample 编写和客户问题定位展开。排查问题时，要能判断问题大概在哪一层：客户调用 SDK 是否正确、SDK 封装逻辑是否有问题、驱动是否正确配置硬件、输入码流是否异常，或者硬件模块输出是否异常。

#### 2. 视频/图像编解码模块在芯片中的位置

视频/图像编解码模块是芯片里的专用硬件加速单元，通常属于多媒体子系统，用来高效处理视频和图片数据。它位于应用、SDK、驱动、硬件这条链路中偏底层的位置：应用通过 SDK 调接口，SDK 通过驱动配置硬件，硬件模块真正完成 H.264/H.265/JPEG 的编码或解码。

硬件模块存在的原因是视频数据量很大，完全靠 CPU 处理会慢、耗电高、占资源。专用硬件可以更快、更省电地把 `.h264`、`.h265` 码流解码成 YUV 图像帧，或把 YUV 图像帧编码成视频码流。理解这个位置后，后续定位问题就可以按应用参数、SDK 接口、驱动配置、输入码流、硬件输出逐层排查。

#### 3. 硬件编解码器、驱动、SDK、应用之间的关系

这几层可以理解成一条调用链：应用负责提出功能需求，SDK 把硬件能力封装成客户容易调用的接口，驱动把 SDK 的请求翻译成硬件寄存器、内存地址、中断等底层配置，真正完成 H.264/H.265/JPEG 编码或解码的是芯片里的硬件编解码器。

例如解码 `.h264` 时，应用调用 SDK 创建解码通道、送入码流、获取输出帧；SDK 通过驱动配置输入码流地址、输出 buffer、分辨率和启动命令；硬件解码器最后输出 YUV 图像帧。编码时方向相反，应用把 YUV 帧交给 SDK，底层硬件编码器压缩成 `.h264` 或 `.h265` 码流。实际排查问题时，要按这条链路判断：调用顺序错、参数错、buffer 地址或大小错、输入码流异常、硬件输出异常，都可能表现成解码失败、花屏、码率不对等现象。

### 学习笔记

#### 1. 视频为什么需要压缩

视频必须压缩，因为原始视频数据量非常大，直接存储、传输和处理都会占用大量带宽、存储空间和算力。例如 1920x1080、30fps 的原始 YUV 视频，每秒就是几十 MB 级别的数据，不压缩很难在网络、磁盘和嵌入式芯片系统里长期使用。

编码器的作用就是利用画面内部和帧与帧之间的冗余，把一帧帧原始图像压成 `.h264`、`.h265` 等码流；解码器再把码流还原成 YUV 图像帧。工作中常见的码率、画质、延迟、花屏、解码失败，本质上都和这条压缩/解压链路有关。

#### 2. 原始视频数据基础

原始视频可以理解为还没经过 H.264/H.265 压缩的一帧帧图像数据，岗位里常见的是 YUV，而不是 RGB。分辨率决定一帧有多少像素，例如 1920x1080；帧率决定每秒有多少帧，例如 30fps；像素格式决定这些像素数据在内存里怎么存放，例如 YUV420、NV12、NV21。

RGB 更接近显示和图像处理里的直观颜色表示，YUV 则把亮度和色度分开，更适合视频压缩。YUV420 表示色度信息做了采样压缩，数据量比 RGB 小；NV12 和 NV21 都是常见的 YUV420 半平面格式，区别主要在 UV 分量排列顺序不同。SDK 或硬件要求的分辨率、帧率、像素格式、buffer 大小如果不匹配，常见结果就是花屏、绿屏、颜色异常或解码/编码失败。

#### 3. 编码和解码的基本流程

视频链路里要先分清两类数据：原始图像帧和压缩码流。原始图像帧通常是 YUV/RGB，数据量大，适合显示、图像处理或送给编码器；压缩码流通常是 H.264/H.265，数据量小，适合存储、传输或送给解码器。编码和解码就是在这两类数据之间转换。

```text
编码：原始 YUV/RGB 图像帧 -> H.264/H.265 压缩码流
解码：H.264/H.265 压缩码流 -> 原始 YUV 图像帧
```

编码器可以理解成“压缩器”。它通常吃进去一帧帧 YUV，比如 NV12 格式的图像，结合分辨率、帧率、像素格式、码率、GOP 等参数，把大体积的原始画面压成 `.h264` 或 `.h265` 码流。解码器可以理解成“还原器”。它吃进去 H.264/H.265 码流，解析里面的参数和帧数据，再输出一帧帧 YUV，后面才能显示、保存图片、做后处理或和 golden 数据对比。

```text
摄像头 / 图像处理模块 / 测试 YUV 文件
        -> 原始 YUV 帧
        -> 编码器
        -> H.264/H.265 码流
        -> 保存文件 / 网络传输 / 客户播放

.h264 / .h265 码流 / 从 mp4 中拆出的视频码流
        -> 解码器
        -> 原始 YUV 帧
        -> 显示 / 后处理 / 保存 / 对比测试
```

| 项目 | 编码 | 解码 |
| --- | --- | --- |
| 输入 | 原始 YUV/RGB 图像帧 | H.264/H.265 压缩码流 |
| 输出 | H.264/H.265 压缩码流 | 原始 YUV 图像帧 |
| 主要目的 | 压缩，便于存储或传输 | 还原，便于显示或处理 |
| 常见参数 | 分辨率、帧率、像素格式、码率、GOP | 码流格式、SPS/PPS、输出 buffer、分辨率 |
| 常见问题 | 码率不对、编码失败、输出码流异常 | 解码失败、花屏、绿屏、丢帧 |

放到 SDK 工作里，应用一般不会直接操作硬件编解码器，而是调用 SDK 创建编码/解码通道、设置参数、送入输入数据、取回输出结果。调试时先问自己：现在是编码还是解码？输入输出方向有没有搞反？例如把 `.h264` 送给编码器、把 YUV 送给解码器，方向就是错的。花屏或失败时，也要沿着方向排查：编码侧看输入 YUV 格式、码率/GOP 参数和输出码流；解码侧看码流完整性、SPS/PPS、输出格式、buffer 大小和 stride。

#### 4. GOP、I 帧、P 帧、B 帧

视频是一帧一帧的画面连续播放。如果每一帧都完整保存，数据量会非常大；但真实视频里前后帧通常很像，例如监控画面里背景不变，只有人或车在移动。编码器会利用这个特点：先保存一张比较完整的画面，后面的帧尽量只记录“相对前后帧发生了什么变化”。

| 概念 | 可以先怎么理解 | 是否能独立解码 | 数据量 | 典型影响 |
| --- | --- | --- | --- | --- |
| I 帧 | 一张比较完整的关键画面 | 基本可以 | 大 | 适合起播、拖动、错误恢复 |
| P 帧 | 参考前面帧，只记录变化 | 不可以 | 中 | 常用来降低码率 |
| B 帧 | 同时参考前后帧，记录更省的数据 | 不可以 | 小 | 压缩率更高，但更复杂、可能增加延迟 |
| GOP | 从一个 I 帧开始的一组连续视频帧 | 不是单帧 | 取决于结构 | 影响码率、延迟、seek 和花屏恢复 |

一个常见 GOP 可以长这样：

```text
I  B  B  P  B  B  P  B  B  P
```

这里的逻辑是：I 帧先给解码器一个可靠起点，P 帧参考前面的 I/P 帧记录变化，B 帧再参考前后帧进一步压缩。不断重复这样的 GOP，就组成完整视频码流。

```text
原始视频帧很多
  -> 每帧完整保存太大
  -> 放入 I 帧作为起点
  -> 用 P/B 帧记录后续变化
  -> 形成一个 GOP
  -> 多个 GOP 组成完整视频
```

GOP 长一些，I 帧间隔远，压缩效率通常更好，但拖动播放、解码启动和出错恢复会慢一些；GOP 短一些，I 帧更多，恢复和 seek 更方便，但码率和文件体积会上升。低延迟场景可能减少甚至不用 B 帧，存储类视频则可能用 B 帧换更高压缩率。以后看到 SDK 里的 `gop size`、`idr interval`、`b frame num`，就要想到它们会影响码率、延迟、解码启动和花屏后的恢复速度。

#### 5. 码流、裸流、封装格式的区别

平时看到的 `.mp4`、`.mkv` 通常不是单纯的视频编码数据，而是一个“容器”。容器里可以放视频码流、音频码流、字幕、时间戳、索引、封面和元信息。硬件视频解码器真正关心的通常不是整个容器，而是容器里拆出来的 H.264/H.265 视频码流。

```text
MP4 / MKV 文件
  -> demuxer 拆封装
  -> H.264 / H.265 视频码流
  -> decoder 解码
  -> YUV 图像帧
```

| 名词 | 可以怎么理解 | 常见例子 | 里面主要有什么 | 岗位里怎么用 |
| --- | --- | --- | --- | --- |
| 码流 | 编码器输出的压缩视频数据 | H.264 码流、H.265 码流 | SPS/PPS、帧数据、NALU 等 | 解码器真正要解析的东西 |
| 裸流 | 只保存视频码流的文件 | `.h264`、`.h265` | 基本只有视频编码数据 | 常用来喂硬件解码器测试 |
| 封装格式 | 装码流和其他信息的文件容器 | `.mp4`、`.mkv` | 视频、音频、字幕、时间戳、索引 | 播放器和文件分发常用 |

所以文件后缀不能简单等同于编码格式：`.mp4` 是封装格式，里面可能装 H.264，也可能装 H.265；`.h264`、`.h265` 更接近编码格式对应的裸码流文件。调试硬件解码器时，常见流程是先用 `ffprobe` 看封装文件里的视频编码格式，再用 FFmpeg 抽出裸流，例如 `ffmpeg -i input.mp4 -c:v copy -an output.h264`。其中 `-c:v copy` 表示不重新编码，只复制视频码流，`-an` 表示不要音频。

需要留意的是，从 `.mp4` 里复制出来的视频码流不一定天然符合所有硬件 sample 的输入格式；H.264 里还会遇到 Annex-B、start code、SPS/PPS 等细节。现阶段先抓住主线：播放器通常吃 `.mp4/.mkv`，硬件解码器通常吃 H.264/H.265 码流，SDK sample 常用 `.h264/.h265` 裸流做测试输入。

#### 6. 常见视频文件和码流

`.h264` 和 `.h265` 通常表示裸视频码流，分别对应 H.264/AVC 和 H.265/HEVC 编码数据，适合拿来直接喂给硬件解码器或做底层码流分析。它们通常没有音频和容器索引，播放时需要播放器或工具能自动识别码流格式；有些情况下还需要码流里带有 SPS/PPS/VPS 等参数信息，否则解码器不知道分辨率、profile 等关键配置。

`.mp4` 和 `.mkv` 是封装文件，不等于某一种编码格式。一个 `.mp4` 里可能装 H.264，也可能装 H.265，还可能带 AAC 音频；`.mkv` 更灵活，也可以装多种视频、音频和字幕轨。岗位中常见流程是用 FFmpeg/ffprobe 查看封装文件信息，确认里面的视频编码格式，再按需要抽取 `.h264` 或 `.h265` 裸流给硬件解码器测试。简单记：`.h264/.h265` 更像“压缩后的视频本体”，`.mp4/.mkv` 更像“把视频、音频、时间戳等打包好的文件盒子”。

### 学习笔记

#### 1. H.264 是什么，主要应用场景

H.264 是一种视频压缩编码标准，可以理解成一套“怎么把原始视频压小、又怎么把它解回来”的规则。编码时，H.264 编码器把一帧帧 YUV 原始图像压成 H.264 码流；解码时，H.264 解码器再把码流还原成 YUV 图像帧。

```text
编码方向：YUV 原始帧 -> H.264 编码器 -> H.264 码流
解码方向：H.264 码流 -> H.264 解码器 -> YUV 图像帧
```

不同厂商可以实现不同的 H.264 编码器/解码器，比如软件里的 x264、FFmpeg，或者芯片里的硬件 H.264 模块，但它们都要遵守 H.264 标准规定的码流结构和解码规则。这个岗位通常不是自己发明 H.264 算法，而是让芯片里的 H.264 硬件模块能通过 SDK 被正确使用：客户送进来的 H.264 能正常解码，测试程序送入的 YUV 能正确编码成 H.264。

| 场景 | H.264 怎么出现 |
| --- | --- |
| 监控摄像头 | 摄像头把画面编码成 H.264，方便录像和网络传输 |
| 手机/相机录像 | 设备把视频压成 H.264，再封装成 `.mp4` |
| 直播/视频会议 | 实时把画面编码成 H.264，通过网络发送 |
| 芯片 SDK 测试 | 用 `.h264` 裸流测试硬件解码，或把 YUV 编码成 H.264 检查输出 |

要注意 `.mp4` 不等于 H.264：`.mp4` 是容器，里面的视频流可能是 H.264，也可能是 H.265 或其他格式。后面学 NALU、SPS/PPS、IDR、Annex-B，就是在看 H.264 码流内部到底怎么组织，解码器为什么能读懂它。

#### 2. H.264 编码基本流程

H.264 编码不是把 YUV 简单打包，而是分析画面内容，尽量用更少的数据描述视频。它的核心思路可以先记成一句话：能预测出来的内容就不完整保存，只保存“预测方式”和“预测错了多少”。

```text
输入 YUV 原始帧
  -> 分块
  -> 预测
  -> 计算残差
  -> 变换和量化
  -> 熵编码
  -> 打包成 NALU
  -> 输出 H.264 码流
```

分块是把一帧画面拆成很多小区域，便于分别选择压缩方式。预测是 H.264 压缩的关键：帧内预测参考当前帧里相邻区域，适合 I 帧；帧间预测参考前面或前后视频帧，适合 P/B 帧。预测后仍然会有误差，真实块和预测块之间的差值叫残差，解码时大致可以理解为“预测值 + 残差 = 还原图像块”。

残差还会继续经过变换、量化和熵编码。量化会把部分细节粗略化甚至丢掉，所以 H.264 多数场景是有损压缩；QP 越大通常压得越狠、码率越低，但画质越容易下降。熵编码可以理解成最后再做一层无损压缩，让码流更小。对 SDK/芯片工作来说，不一定要自己实现这些算法，但要知道码率、QP、GOP、输入像素格式、buffer 配置都会影响编码结果和画质。

#### 3. NALU 概念

NALU 是 H.264 码流的基本组织单位，可以先理解成“H.264 码流里一小段一小段的数据包”。H.264 码流不是一整块没有边界的数据，而是由许多 NALU 依次组成，每个 NALU 有自己的类型和内容。

```text
H.264 码流
  -> NALU
  -> NALU
  -> NALU
  -> ...
```

有些 NALU 装的是解码参数，比如 SPS、PPS；有些装的是具体帧数据，比如 IDR slice、普通 slice；还有些可能装 SEI 这类补充信息。每个 NALU 开头有 NAL header，里面会标明这个 NALU 的类型。入门阶段常见类型可以先记这些：

| nal_unit_type | 含义 | 入门理解 |
| --- | --- | --- |
| 1 | 非 IDR slice | 普通 P/B 帧相关数据常见在这里 |
| 5 | IDR slice | 关键帧数据，适合解码重新开始 |
| 6 | SEI | 补充增强信息 |
| 7 | SPS | 序列参数集 |
| 8 | PPS | 图像参数集 |

工作中懂 NALU 很有用，因为硬件解码器实际解析的是一段段 H.264 码流。解码失败、第一帧不出、花屏时，经常要先判断输入是否能正确切成 NALU、是否有 SPS/PPS、是否有 IDR、送给硬件的 buffer 边界是否正确。

#### 4. SPS / PPS / IDR 帧

SPS、PPS、IDR 都是 H.264 解码里很关键的内容。解码器拿到视频帧数据之前，必须先知道这段视频的基本规则，例如宽高、profile、level、参考帧数量、某些图像级编码参数等；这些规则主要靠 SPS 和 PPS 提供。

| 概念 | NALU 类型 | 可以怎么理解 | 缺失/异常时可能现象 |
| --- | --- | --- | --- |
| SPS | 7 | 视频序列的总说明书，偏全局规则 | 解码器不知道宽高/规则，可能初始化失败 |
| PPS | 8 | 图像级解码说明，依赖 SPS | slice 无法正确解释，可能解码失败或花屏 |
| IDR | 5 | 可靠重新开始点，一种特殊关键帧 | 起播、seek、错误恢复可能异常 |
| 普通 slice | 1 | 普通帧数据 | 丢失可能导致花屏、丢帧 |

SPS 更偏“一段视频整体怎么解”，PPS 更偏“某些图像或 slice 相关参数怎么解”。IDR 可以理解成很强的关键帧：普通 I 帧主要是不依赖其他帧来解当前图像，而 IDR 还会刷新解码器参考状态，告诉解码器后面的帧不会再依赖 IDR 之前的旧帧。因此起播、seek、花屏恢复时，IDR 特别重要。

常见 `.h264` 裸流开头可能是：

```text
start code + SPS
start code + PPS
start code + IDR
start code + P frame
...
```

排查 H.264 硬件解码失败时，常见第一步就是看码流前面有没有 SPS/PPS，起始位置附近有没有 IDR，硬件 sample 是否要求每个 IDR 前重复携带 SPS/PPS。

#### 5. Annex-B 格式和 start code

Annex-B 是 H.264 裸流里很常见的一种组织方式。它的特点是每个 NALU 前面用 start code 做分隔，常见 start code 是 `00 00 01` 或 `00 00 00 01`。start code 的作用很朴素：告诉解析器“新的 NALU 从这里开始”。

```text
00 00 00 01 [SPS]
00 00 00 01 [PPS]
00 00 00 01 [IDR]
00 00 00 01 [P slice]
```

这和 `.mp4` 容器内部常见的 H.264 存储方式不完全一样。`.mp4` 里经常不是用 start code 分隔 NALU，而是在每个 NALU 前面写一个长度字段，告诉解析器后面这个 NALU 有多长。

| 格式 | NALU 怎么分隔 | 常见位置 | 调试关注 |
| --- | --- | --- | --- |
| Annex-B | `00 00 01` 或 `00 00 00 01` | `.h264` 裸流、传输流、硬件测试输入 | start code 是否存在，SPS/PPS 是否在前 |
| MP4 内部格式 | 长度字段 | `.mp4` 容器内部 | 抽流后可能需要转 Annex-B |

所以从 `.mp4` 里直接复制出来的视频流，不一定符合某些硬件 sample 的输入要求。如果 sample 期待 Annex-B，通常需要用 FFmpeg 的 bitstream filter 转换，例如 `ffmpeg -i input.mp4 -c:v copy -bsf:v h264_mp4toannexb output.h264`。先不必死背命令，重点记住：硬件解码 sample 常常更喜欢 Annex-B，因为它容易按 start code 找到一个个 NALU。

#### 6. I/P/B 帧在 H.264 中的作用

H.264 也使用 I/P/B 帧来在画质、码率和延迟之间做平衡。I 帧主要使用帧内预测，更依赖当前帧内部的信息，不依赖其他帧来预测画面，所以独立性强，但数据量大。IDR 通常是一种特殊的 I 帧，不过不是所有 I 帧都等同于 IDR；入门阶段可以先记住：IDR 比普通 I 帧更适合作为解码重新开始点。

P 帧使用帧间预测，参考前面的帧，只记录相对参考帧发生的变化，所以数据量比 I 帧小。B 帧可以参考前后帧，压缩效率通常更高，但解码和显示顺序更复杂，也可能增加延迟。实时低延迟场景可能减少或不用 B 帧，存储类视频更可能用 B 帧换更高压缩率。

| 帧类型 | 主要参考 | 特点 | 常见影响 |
| --- | --- | --- | --- |
| I 帧 | 当前帧内部 | 独立性强，数据量大 | 起播、seek、错误恢复更方便 |
| P 帧 | 前面的参考帧 | 记录变化，数据量中等 | 降低码率，依赖参考帧 |
| B 帧 | 前后参考帧 | 压缩更强，复杂度更高 | 码率更低，但可能增加延迟 |

把第 3 到第 6 点串起来看：H.264 码流由很多 NALU 组成；Annex-B 用 start code 把 NALU 分隔开；SPS/PPS 这类 NALU 提供解码说明；IDR/I/P/B 相关 slice NALU 提供具体画面数据。解码器按这些规则和数据，最后把 H.264 码流还原成 YUV 图像帧。

#### 7. H.264 解码时常见问题

H.264 解码问题不要一上来就判断“硬件坏了”。更实用的思路是先沿着输入链路排查：送进去的是不是 H.264 码流，码流格式是不是 sample 要求的 Annex-B，能不能正常切出 NALU，前面有没有 SPS/PPS，起始位置附近有没有 IDR，输出 YUV 的格式、stride 和 buffer 大小是否按硬件要求处理。

```text
解码异常
  -> 输入是不是 H.264，而不是整个 .mp4？
  -> 是否符合 sample 要求的 Annex-B / start code 格式？
  -> 能不能找到 SPS/PPS/IDR？
  -> 送入 buffer 的边界、长度、顺序是否正确？
  -> 输出 YUV 格式、stride、buffer 大小是否匹配？
```

| 现象 | 常见原因 | 排查方向 |
| --- | --- | --- |
| 花屏 | 码流中间丢数据、参考帧缺失、P/B 帧依赖的帧没送到、输出 YUV 格式或 stride 理解错 | 先用 FFmpeg 软解验证码流，再看送帧顺序、buffer 边界、输出格式 |
| 绿屏/颜色异常 | 解码可能成功了，但显示侧按错像素格式解释 YUV，例如 NV12/NV21 搞反，或 stride/地址偏移不对 | 检查硬件输出格式、Y/UV 平面地址、stride、显示程序的像素格式 |
| 丢帧/卡顿 | 输入码流不连续、送帧速度不稳定、buffer 不够、解码器返回错误未处理、B 帧导致显示顺序复杂 | 看 SDK 返回值、时间戳/帧序、输入是否完整、输出队列是否及时取帧 |
| 解码失败 | 输入不是裸 H.264、Annex-B 格式不符合、SPS/PPS 缺失、profile/level 或分辨率超出硬件支持 | 用 `ffprobe` 确认编码格式，用工具检查 NALU 类型和硬件支持范围 |
| SPS/PPS 缺失 | 从 `.mp4` 抽流方式不对，或只截取了中间 P/B 帧，导致解码器缺少规则说明 | 确认码流开头或 IDR 前是否有 SPS/PPS，必要时重新抽流或补参数 |

排查时可以把“软解”和“硬解”结果对比起来看：如果 FFmpeg 软解也失败，优先怀疑输入码流本身；如果软解正常、硬解失败，再重点看硬件支持范围、输入格式、Annex-B/start code、buffer 对齐和 SDK 调用顺序。如果硬解输出 YUV 正常但显示花屏，问题可能不在解码，而在显示侧按错了像素格式或 stride。

这一节和前面的概念是连起来的：NALU 决定码流能不能被正确切分，SPS/PPS 决定解码器是否知道规则，IDR 决定能否可靠起播和恢复，I/P/B 帧的参考关系决定丢帧后为什么可能花屏。实际工作中，把这些概念串成排查路径，比单独背名词更重要。

#### 1. H.265 和 H.264 的关系

H.265 是 H.264 之后的新一代视频压缩编码标准。两者解决的是同一类问题：原始 YUV 视频数据太大，需要通过编码器压缩成码流，便于存储、传输和播放；解码时再把码流还原成 YUV 图像帧。

```text
编码：YUV 原始帧 -> H.264/H.265 编码器 -> H.264/H.265 码流
解码：H.264/H.265 码流 -> H.264/H.265 解码器 -> YUV 图像帧
```

H.264 又叫 AVC，H.265 又叫 HEVC。从名字看，HEVC 的重点就是更高压缩效率。可以把它们理解成两代“视频压缩语言”：H.264 更早、更主流、兼容性很强；H.265 更新、更复杂，目标是在接近画质下用更低码率。

| 标准 | 另一个名字 | 可以怎么理解 |
| --- | --- | --- |
| H.264 | AVC | 老一代但仍非常主流，兼容性强 |
| H.265 | HEVC | 新一代压缩效率更高，更适合高清/超高清视频 |

它们都不是容器格式。`.mp4` 里可以装 H.264，也可以装 H.265；`.h264` 通常表示 H.264 裸流，`.h265` 通常表示 H.265 裸流。对 SDK/芯片岗位来说，H.264 和 H.265 往往是同一类工作对象：创建编码/解码通道时选择不同 codec type，底层可能走不同硬件模块、不同码流解析流程和不同参数集。

#### 2. H.265 相比 H.264 的优势

H.265 最核心的优势是：同等画质下通常比 H.264 需要更低码率；反过来说，同样码率下 H.265 往往能保留更好的画质。具体节省多少取决于内容复杂度、分辨率、编码器实现和参数，不要机械记固定比例，但在高清、4K、监控存储、带宽敏感场景里，H.265 的压缩效率优势通常更明显。

H.265 更省码率，主要因为它的压缩工具更复杂、更精细。它的块划分更灵活，可以用大块描述天空、墙面这类平坦区域，用小块处理人物边缘、文字、树叶等细节区域；它的帧内/帧间预测能力也更强，预测越准，残差越小，需要保存的数据就越少。

```text
预测越准
  -> 残差越小
  -> 需要编码的数据越少
  -> 码率更低
```

但 H.265 不是只有优点，它的代价是复杂度更高、兼容性不如 H.264。软件编码 H.265 往往更慢，硬件解码需要芯片有对应模块，老设备、老浏览器或老播放器可能不支持。实际项目中选择 H.264 还是 H.265，通常是在压缩效率、兼容性、硬件能力、功耗和延迟之间做取舍。

| 对比点 | H.264 | H.265 |
| --- | --- | --- |
| 压缩效率 | 较高，成熟稳定 | 更高，同等画质码率更低 |
| 兼容性 | 非常好，设备支持广 | 不如 H.264，老设备可能不支持 |
| 复杂度 | 相对低 | 更高 |
| 适合场景 | 通用播放、直播、兼容性优先 | 高清/4K、监控存储、带宽敏感 |
| 岗位关注 | 是否能正常编解码、SPS/PPS、Annex-B | 是否支持 H.265、VPS/SPS/PPS、分辨率/帧率/profile/level 限制 |

工作里看到 H.265，要习惯继续确认：芯片是否支持 H.265 编码或解码，支持到什么分辨率和帧率，支持哪些 profile/level，输入是 `.h265` 裸流还是 `.mp4` 里的 H.265 视频流，是否带 VPS/SPS/PPS，客户播放环境是否支持 H.265。简单记：H.264 胜在兼容性，H.265 胜在压缩效率。
