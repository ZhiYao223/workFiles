### 学习笔记

#### 1. 芯片多媒体系统软件岗位是做什么的

这个岗位的核心不是单纯写 App，也不是单纯做视频算法，而是让芯片里的视频/图像编解码硬件模块能被软件正确使用起来。芯片中会有视频编码器、视频解码器、JPEG 编解码器等硬件模块，软件需要配置输入输出地址、分辨率、编码格式、码率、启动停止等参数，再通过 SDK 封装成客户能调用的接口。

实际工作会围绕 FPGA 验证、回片验证、SDK 开发维护、Demo/Sample 编写和客户问题定位展开。排查问题时，要能判断问题大概在哪一层：客户调用 SDK 是否正确、SDK 封装逻辑是否有问题、驱动是否正确配置硬件、输入码流是否异常，或者硬件模块输出是否异常。

#### 2. 视频/图像编解码模块在芯片中的位置

视频/图像编解码模块是芯片里的专用硬件加速单元，通常属于多媒体子系统，用来高效处理视频和图片数据。它位于应用、SDK、驱动、硬件这条链路中偏底层的位置：应用通过 SDK 调接口，SDK 通过驱动配置硬件，硬件模块真正完成 H.264/H.265/JPEG 的编码或解码。

硬件模块存在的原因是视频数据量很大，完全靠 CPU 处理会慢、耗电高、占资源。专用硬件可以更快、更省电地把 `.h264`、`.h265` 码流解码成 YUV 图像帧，或把 YUV 图像帧编码成视频码流。理解这个位置后，后续定位问题就可以按应用参数、SDK 接口、驱动配置、输入码流、硬件输出逐层排查。

#### 3. 硬件编解码器、驱动、SDK、应用之间的关系

这几层可以理解成一条调用链：应用负责提出功能需求，SDK 把硬件能力封装成客户容易调用的接口，驱动把 SDK 的请求翻译成硬件寄存器、内存地址、中断等底层配置，真正完成 H.264/H.265/JPEG 编码或解码的是芯片里的硬件编解码器。

例如解码 `.h264` 时，应用调用 SDK 创建解码通道、送入码流、获取输出帧；SDK 通过驱动配置输入码流地址、输出 buffer、分辨率和启动命令；硬件解码器最后输出 YUV 图像帧。编码时方向相反，应用把 YUV 帧交给 SDK，底层硬件编码器压缩成 `.h264` 或 `.h265` 码流。实际排查问题时，要按这条链路判断：调用顺序错、参数错、buffer 地址或大小错、输入码流异常、硬件输出异常，都可能表现成解码失败、花屏、码率不对等现象。

### 学习笔记

#### 1. 视频为什么需要压缩

视频必须压缩，因为原始视频数据量非常大，直接存储、传输和处理都会占用大量带宽、存储空间和算力。例如 1920x1080、30fps 的原始 YUV 视频，每秒就是几十 MB 级别的数据，不压缩很难在网络、磁盘和嵌入式芯片系统里长期使用。

编码器的作用就是利用画面内部和帧与帧之间的冗余，把一帧帧原始图像压成 `.h264`、`.h265` 等码流；解码器再把码流还原成 YUV 图像帧。工作中常见的码率、画质、延迟、花屏、解码失败，本质上都和这条压缩/解压链路有关。

#### 2. 原始视频数据基础

原始视频可以理解为还没经过 H.264/H.265 压缩的一帧帧图像数据，岗位里常见的是 YUV，而不是 RGB。分辨率决定一帧有多少像素，例如 1920x1080；帧率决定每秒有多少帧，例如 30fps；像素格式决定这些像素数据在内存里怎么存放，例如 YUV420、NV12、NV21。

RGB 更接近显示和图像处理里的直观颜色表示，YUV 则把亮度和色度分开，更适合视频压缩。YUV420 表示色度信息做了采样压缩，数据量比 RGB 小；NV12 和 NV21 都是常见的 YUV420 半平面格式，区别主要在 UV 分量排列顺序不同。SDK 或硬件要求的分辨率、帧率、像素格式、buffer 大小如果不匹配，常见结果就是花屏、绿屏、颜色异常或解码/编码失败。

#### 3. 编码和解码的基本流程

编码可以理解为把原始图像帧压缩成码流。应用或测试程序先准备一帧帧 YUV 数据，SDK 把分辨率、帧率、像素格式、码率、GOP 等参数传给底层编码器，硬件编码器再利用帧内压缩和帧间压缩，把这些 YUV 帧变成 `.h264`、`.h265` 这样的压缩码流。帧内压缩主要利用一张图内部相邻区域的相似性，帧间压缩主要利用前后帧之间的相似性，例如画面里背景不变、只有人物或车辆在移动。

解码是反方向：输入端送入 H.264/H.265 码流，解码器解析码流里的参数和帧数据，按编码时记录的信息恢复出一帧帧 YUV 图像。岗位里要特别注意“输入”和“输出”的类型：编码器通常输入 YUV、输出压缩码流；解码器通常输入压缩码流、输出 YUV。很多调试问题都可以先按这个方向判断，例如编码失败要看输入 YUV 格式和编码参数，解码失败要看码流是否完整、SPS/PPS 是否存在、输出 buffer 是否够大。

#### 4. GOP、I 帧、P 帧、B 帧

GOP 可以理解为一组连续的视频帧，通常从一个 I 帧开始，后面跟着若干 P 帧或 B 帧。I 帧也叫关键帧，它基本可以独立解码，不依赖前后帧，代价是数据量比较大；P 帧会参考前面的帧，只记录相对变化，数据量更小；B 帧可以同时参考前后帧，压缩效率更高，但会增加编码、解码和显示顺序处理的复杂度，也可能带来更多延迟。

实际看视频时，画面是连续的，但码流里并不是每一帧都完整保存一张图片。编码器会周期性放入 I 帧，中间用 P/B 帧记录变化，这样才能大幅降低码率。GOP 长一些，压缩效率通常更好，但随机 seek、出错恢复会更慢；GOP 短一些，关键帧更多，文件或码率会变大，但更适合低延迟、快速切换和错误恢复。做 SDK 或芯片验证时，GOP size、是否有 B 帧、IDR/I 帧间隔都会影响码率、延迟、解码启动和花屏恢复。

#### 5. 码流、裸流、封装格式的区别

码流一般指编码器输出的压缩视频数据，例如 H.264/H.265 的一串 NALU。裸流是只包含编码数据本身的文件，常见后缀是 `.h264`、`.h265`，里面通常没有完整的文件容器信息，也不负责保存音频、字幕、封面、时间轴等内容。裸流更接近硬件编解码器真正关心的输入输出，所以在芯片验证、驱动调试、SDK sample 里很常见。

封装格式则像一个“文件容器”，常见 `.mp4`、`.mkv`，里面可以装视频码流、音频码流、字幕、时间戳、索引等信息。播放器播放 `.mp4` 时，通常先 demux 拆出里面的 H.264/H.265 视频码流，再交给解码器解码成 YUV。调试时要分清楚：硬件解码器一般不直接理解 `.mp4` 容器，它要的是从容器里取出来的视频码流；如果把整个 `.mp4` 文件当成 H.264 裸流送给硬件，通常会解析失败。

#### 6. 常见视频文件和码流

`.h264` 和 `.h265` 通常表示裸视频码流，分别对应 H.264/AVC 和 H.265/HEVC 编码数据，适合拿来直接喂给硬件解码器或做底层码流分析。它们通常没有音频和容器索引，播放时需要播放器或工具能自动识别码流格式；有些情况下还需要码流里带有 SPS/PPS/VPS 等参数信息，否则解码器不知道分辨率、profile 等关键配置。

`.mp4` 和 `.mkv` 是封装文件，不等于某一种编码格式。一个 `.mp4` 里可能装 H.264，也可能装 H.265，还可能带 AAC 音频；`.mkv` 更灵活，也可以装多种视频、音频和字幕轨。岗位中常见流程是用 FFmpeg/ffprobe 查看封装文件信息，确认里面的视频编码格式，再按需要抽取 `.h264` 或 `.h265` 裸流给硬件解码器测试。简单记：`.h264/.h265` 更像“压缩后的视频本体”，`.mp4/.mkv` 更像“把视频、音频、时间戳等打包好的文件盒子”。
